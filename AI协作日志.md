# AI 协作日志/项目主要思考过程

> **关于本文档的说明**
> 
> 本人与 AI 的协作模式为"深度融合式迭代"，而非"结构化咨询"。过程中大量代码、分析和假设由 AI 生成，但所有关键决策（是否采纳、如何验证、何时推翻）由实验结果和代码审计驱动。由于认知风格偏向语义记忆，过程性对话细节未被逐句保留，但关键决策节点均有代码与结果作为证据。

---

## 一、协作模式

如果把常见的 AI 协作模式分为两类：

- **半监督模式**（他人常见）：AI 作为"PHM 专家"给出执行计划，人负责评估和按部就班执行。
- **自监督模式**（本人采用）：AI 作为样本/假设生成器，人通过实验构造监督信号，双方共同被结果教育。

在本项目中，我并未让 AI 替我制定计划，而是**先自己定义问题**（"这道题能不能做成 benchmark？"），然后让 AI 帮我填充执行细节。当 AI 给出解释时，我的第一反应不是"听起来对不对"，而是**"跑个实验看看"**。

## 二、项目演进：从"做题"到"审计"

### 2.1 初始意图：拉个榜，塞个 ResNet

最初看到题目时，我的直觉是"做个二分类复现，顺便塞个我喜欢的 ResNet-1D 进去"。但在搜索过程中发现：社区已经有大量复现，且质量参差不齐——有人准确率虚高到 76%+（后诊断为划分错误），有人只跑单模型无对比，有人用滑动窗口把航班级任务偷换成片段级。

**决策点**：与其重复造轮子，不如建立一个可验证的评估基础设施，把现有的复现结果审一遍。

**AI 的角色**：帮我整理了搜索到的复现链接，但"做成 benchmark"这个方向是我定的。

### 2.2 建立标准：比的是架构，不是预算

在审阅社区复现时，发现很多人在做数据增强，调参（相比数据集论文的默认设置）和消融。如果全部收录，benchmark 会无限膨胀，而且失去比较的意义。

**决策点**：制定收录原则，明确不收录调参版，消融版，并且不允许任何形式的数据增强，只收录结构性修改（如换骨干网络、换特征提取阶段）。核心原则是：**benchmark 比的是架构本身，不是预算。**

**AI 的角色**：帮我起草了协议文本，但"什么该收录，什么不该"的判定标准是我基于审计经验逐条确定的。

### 2.3 审计发现 1：MiniRocket 官方代码的维度 Bug

这是整个项目中最关键的发现。

**背景**：官方 Colab 报 MiniRocket 准确率 59.8%，但我的 sktime 实现跑到 72%。AI 最初推测是"tsai 的 NN 头 vs sktime 的 Ridge 分类器"导致差异。

**我的质疑**：后来发现 tsai 版我自己跑也是 72%，感觉很奇怪，于是让ai帮我看我的脚本哪里错了（结果没想到我的是正确的，反而是官方代码错了）。

**验证过程**：
1. 下载官方完整 `.py` 让ai逐行审计，发现官方未对输入维度转置：`MiniRocketFeatures(train_X.shape[1], train_X.shape[2])` 直接接收了 `(N, 4096, 23)`，导致 `c_in=4096, seq_len=23`。
2. ai建议我在自己的 tsai 实现中**故意注释掉 `np.transpose`**，复刻官方配置，结果 0.5895，与官方 59.8% 几乎一致。
3. 修正后回到 0.7318，确认差距约 13 个百分点。

**结论**：官方代码以及很多复现者复制了同一错误。交叉验证在此案例中不仅未发现问题，反而成了错误传播的放大器。

**AI 的角色**：在我要求下帮我对比了 tsai 和 sktime 的 API 设计差异，review代码，以及提出验证建议。

### 2.4 审计发现 2：BiMamba 的"虚假繁荣"

SongX-1 的 README 声称 CNN+BiMamba 达到 81%，看起来很唬人。

**我的质疑**：双向 Mamba 从 NLP 迁移到多变量时序，真的有用吗？

**验证过程**：
1. 按 AMBC 标准协议（禁止数据增强）复现，结果 CNN-only 0.73，BiMamba 0.68，F1 标准差爆炸到 0.11。
2. 分析发现原作者用 Sliding Window 把航班级任务转为片段级，训练数据量膨胀，评估分母变了。
3. 统一协议后，BiMamba 不仅没提升，反而被纯 CNN 全面碾压。

**结论**：跨领域迁移不是换模块名字那么简单。NLP 里的双向架构在时序里不一定 work。

**AI 的角色**：帮我写了复现代码，但" Sliding Window 是否公平"的判断和最终决策是我做的。

### 2.5 硬件兼容性：TF 到 PyTorch 的被迫迁移

原版基准是 TensorFlow + Colab TPU v2-8。本地 RTX 5060 Ti（Blackwell / sm_120）上，TF不兼容，换WSL（linux）也不行。

**决策点**：直接重写 PyTorch 版。

**后续问题**：ConvMHSA 官方 batch_size=128 在 16GB 显存下吃到 97.8%，触发 allocator thrashing，单 epoch 400 秒（23 小时/fold）。

**解决方案**：ai建议我加上梯度累积等工程优化，我在评估发现几乎不影响结果之后采用了，最终显存降到 60%。代码和结果 JSON 中打 `[HARDWARE APPROXIMATION]` 标签。

**AI 的角色**：帮我写了 PyTorch 移植代码和梯度累积逻辑，但"是否接受近似"以及"如何标记 audit trail"是我决定的。

### 2.6 数据集审计：不仅审模型，也审数据集

第一阶段收尾时，所有模型在二分类上都卡在 0.71-0.76。

**我的推断**：我问ai为什么会这样，ai告诉我`P(RUL>2days)` 这种粗糙的 yes/no 标签，把慢性磨损的渐变问题硬压成二元判断，真正的故障信号被飞行员操作噪声淹没，这个任务上限就这样了。

**结论**：NGAFID 维护预测任务在"准确率"维度上信息熵很低，但在"效率"维度上仍有价值。AMBC 的最大收获不是填完了表，而是证明了"什么样的数据集值得做成benchmark"。

**AI 的角色**：帮我整理了对比表格和 Pareto 图，但"数据集本身不适合长期 benchmark 化"这个结论是我基于证据链推导的。

## 三、元认知：我学到了什么

### 3.1 关于 Benchmarking

- **70% 工作量在数据对齐**：确保每个人的预处理、划分、评估口径一致。
- **交叉验证一致 ≠ 实现正确**：复现者们都跑出 59.8%，不代表 59.8% 是对的。
- **Benchmark 也有生命周期**：出生、被引用、被遗忘。不是所有数据集都适合benchmark。

### 3.2 关于 AI 协作

- **AI 是外接硬盘，不是导师**：我负责定义问题，AI 负责生成候选假设和代码。
- **实验结果是唯一的 ground truth**：当 AI 的解释和实验结果冲突时，相信实验。
- **事后归因是我的认知风格**：过程性细节会遗忘，但结果和结论会沉淀。

### 3.3 关于问题驱动

我的所有项目都遵循同一个公式：

> **"我想做 Z，需要哪些 X 和 Y？现学来得及吗？"**

不是"我掌握了 X，能在 Y 场景做什么"，而是"我有一个问题，然后拉着 AI 一起把坑填上"。AMBC 只是这个模式的又一次实例化。
